# zkm-photo

zkm plugin: imports photos (JPEG today; PNG/TIFF/HEIC on the roadmap) into the
knowledge store. EXIF (date, GPS decimal degrees, camera model, dimensions) →
frontmatter; raw bytes → CAS; canonical symlink + multi-producer sidecar in
`inbox/photos/`.

Independent git repo living at `~/src/zkm/plugins/zkm-photo/` inside the zkm
core checkout (core gitignores `plugins/`). See `ARCHITECTURE.md` for design
decisions with rationale; `ROADMAP.md` for the executor task queue.

## Commands

```bash
uv sync --extra dev          # env: zkm core editable from ../.. + pytest/ruff/Pillow/piexif
uv run pytest                # hermetic suite (tmp_path stores, synthetic fixtures, no network)
uv run pytest -k <expr>      # one test / one roadmap item's done-check
uv run ruff check <files>    # lint (line-length 100, py311); keep files YOU touch clean
uv run python tests/build_fixtures.py   # regenerate committed fixture images (deterministic)
```

Run via core: `ZKM_STORE=/tmp/kb uv run zkm convert photo` from the core repo
(requires `source_dir` under the `photo:` key in `$ZKM_STORE/zkm-config.yaml`).

## Layout

```
convert.py                  # filesystem-discovery SHIM only — re-exports zkm_photo.convert.convert
plugin.yaml                 # manifest (root copy, used by filesystem discovery)
src/zkm_photo/
├── __init__.py
├── convert.py              # the actual implementation
└── plugin.yaml             # manifest (wheel copy)
tests/
├── build_fixtures.py       # synthetic fixture generator (Pillow + piexif, dev-only)
├── fixtures/*.jpg|png|tif|heic   # committed binaries — NEVER real photos
└── test_convert.py         # main suite
```

## Conventions

- Config keys are **bare snake_case** read from `zkm-config.yaml` (`source_dir`),
  per zkm core M2. Never reintroduce env-var style keys (`PHOTO_SOURCE_DIR` is
  the pre-M2 form and was removed).
- `name: photo` (bare handle, no `zkm-` prefix) per core plugin spec.
- Object storage is **delegated to core**: `zkm.atomic`, `zkm.cas`,
  `zkm.hashing`, `zkm.inbox`. Do not reimplement; see core
  `docs/object-storage.md`.
- Frontmatter must satisfy core `zkm.conformance.FRONTMATTER_REQUIRED`
  (source, date, tags, sha256, processor, processor_version).
- Conventional commits; loose-0.x versioning (patch = bugfix, minor = anything
  else); every pyproject version bump is tagged `vX.Y.Z` in the same commit.
- `uv.lock` is committed (polyrepo convention).

## Gotchas

- **Two `plugin.yaml` copies** (root + `src/zkm_photo/`) must stay byte-identical:
  root serves filesystem discovery, the package copy ships in the wheel. Update
  both or discovery and wheel installs disagree.
- **Root `convert.py` is a shim** — tests import `zkm_photo.convert`; the shim
  exists only so core's filesystem discovery (`_inject_plugin_venv` adds `src/`
  to sys.path) can load the plugin without an install. Don't add logic there.
- **Fixtures are synthetic** — generated by `tests/build_fixtures.py`; the
  generator is deterministic (regen leaves committed files byte-identical).
  Regenerate and commit when adding fixtures; never commit real photographs.
- **Suite drift history**: the SB5 repackage (flat → `src/zkm_photo/`) and M2
  config migration once landed without updating tests, leaving the suite
  uncollectable at HEAD. Always run `uv run pytest` after touching packaging or
  config plumbing.
- **EXIF dates are naive** ("YYYY:MM:DD HH:MM:SS", no tz) while core
  conformance requires tz-aware ISO 8601 — known gap, roadmap:33e5. The mtime
  fallback path is already tz-aware.
- Core's dirty-tree guard applies when invoked via `zkm convert photo`:
  uncommitted tracked changes in this repo refuse to run
  (`ZKM_BYPASS_DIRTY_CHECK=1` to bypass in dev).
- exifread ≥3.5 parses JPEG/TIFF/PNG-eXIf/HEIC/WebP; `_parse_exif` wraps it in
  a broad try/except so corrupt or unsupported EXIF degrades to `{}` (mtime
  date fallback) instead of crashing.
